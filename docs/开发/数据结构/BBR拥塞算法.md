## BBR概念
BBR（Bottleneck Bandwidth and Round-trip propagation time）是Google开发的一种TCP拥塞控制算法，用于提升网络吞吐量并降低延迟
## BBR诞生的原因
传统的拥塞控制算法（基于丢包）有一个致命弱点：它们把“丢包”等同于“拥塞”。
```
但在现代高速网络或长距离链路中，丢包可能是因为链路质量不好，而不是网络堵塞。
传统算法一旦丢包就大幅减速，导致带宽利用率极低。
```
BBR 的逻辑是： 我不看丢包，我只看两个物理指标：
```
RTprop (Round-trip propagation time)： 链路的固有延迟（不排队时的往返时间）。
BtlBw (Bottleneck Bandwidth)： 瓶颈带宽（这条路最宽的地方能跑多快）。
```
## BBR是如何计算的
```
BDP = BtlBw × RTprop
```
## 运行流程
### 1.Startup (启动阶段)
指数级增加发送速率，直到发现带宽不再增长（找到 BtlBw）。
### 2.Drain (排空阶段)
把刚才启动阶段攒下的排队缓存“排”出去。
### 3.ProbeBW (探测带宽)
这是最常态的情况。它会周期性地稍微多发一点点（1.25 倍速），看看路有没有变宽；如果没变，就减速降回来。
### 4.ProbeRTT (探测延迟)
每隔一段时间（约 10 秒），大幅减速，清空所有排队，测一下现在最快的物理延迟是多少。
## 案例：跨国专线
假设你有一条从北京到旧金山的链路： 
物理带宽上限：100 Mbps，
链路固有延迟 (RTprop)：150 ms（单向 75 ms）
### 计算
1. 将带宽单位转换为字节/秒：100 Mbps=12.5 MB/s。
2. 将延迟转换为秒：150 ms=0.15 s。
3. BDP=12.5 MB/s×0.15 s=1.875 MB。
这意味着只要你在网络中发送的数据总量（In-flight data）不超过 1.875 MB，数据包就不会在路由器里排队，延迟依然保持在 150 ms。
